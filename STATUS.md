Smoke-tested 8 cities (Kochi, Chennai, Jakarta, Cairo, Riyadh, NYC, London, Makkah) against NOAA's online calculator — all within 0.5°. 3. Interference detector tightened — now flags BOTH weak field (<20 μT) AND strong field (>70 μT), not just weak. Strong-field is the masjid scenario (steel beams, AC ducts, speaker magnets). The "needs calibration" banner now fires correctly when Mohammed (or any user) is standing inside a metal-structured building. Outcome: Qibla bearing now sub-degree accurate worldwide (was 2°–17° off depending on region). The hadith "between east and west is qibla" gives ±45° tolerance so users weren't praying invalidly under the old code, but they were facing slightly off the true direction — and they deserved better. Learnings entry "qibla-dipole-model-vs-wmm" documents the root cause + a CI-gate follow-up (vendored NOAA reference fixtures asserting our declination at 5 cities matches within 1°).

- 2026-05-13 — Mushaf v2 renderer fix: code_v2 was being sent as wrong parameter type
  Mohammed installed preview build `69eba95d` (first APK with QF creds baked in), flipped the v2 toggle, and screenshotted five pages — every one rendered as broken Arabic. Disconnected letters on pages 602–604, font specimen shapes on page 1, nothing looked like Quran. The v2 path was clearly active (different layout from v1) but the text was wrong. Root cause: `mushafCache.cacheChapter()` called `fetchQfVerses(N, { words: true, fields: ['code_v2'] })`. QF's `code_v2` is a **word-level** field, requested via `word_fields=`, not a **verse-level** field requested via `fields=`. The wrong parameter was silently ignored; QF returned words with only their default Unicode `text`; our `readCodeV2()` helper fell back to that Unicode; the cache stored Uthmani Arabic in the `code_v2` SQLite column; the renderer fed that Unicode into the QCF_P{N}_V2 font (which expects private-use-area glyph codes); the font's fallback glyphs produced the disconnected-Arabic output Mohammed photographed. Bug had been latent since Stage 2 of v1.8.0 shipped two days earlier (commit `cb4a2b9`). Diagnosed by re-reading the QF docs against today's research brief. Four-part fix: 1. `fetchQfVerses` now exposes a proper `wordFields` parameter that maps to `word_fields` on the wire. JSDoc on the old `fields` option was wrong (listed `code_v2` as valid); fixed. 2. `cacheChapter` now passes `wordFields: ['code_v2', 'line_number', 'page_number', 'position', 'char_type_name']`. Verified locally via `node` script that QF returns proper PUA codes (e.g. `code_v2="ﱁ"` for Al-Ikhlas word 1, page 604 line 3). 3. `readCodeV2()` no longer falls back to `word.text`. The previous defensive shim was the bug amplifier — same anti-pattern as the silent-Mishary fallback fixed yesterday. If QF doesn't return an explicit `code_v2`, we return null, skip the row, and the renderer falls through to the v1 path which works correctly. Loud failure beats silent wrong output. 4. New `migrateMushafCacheIfStaleFormat()` runs on every app start (called from `initializeDatabase()`). Samples one row's `code_v2`; if any codepoint is in the regular Arabic Unicode range 0x0600–0x06FF (which never appears in valid PUA glyph codes), wipes both `mushaf_words` and `mushaf_chapter_cache`. Idempotent. Cleans up the broken cache on Mohammed's device so the next render fetches correctly. Verification: `npx tsc --noEmit` clean. The QF API call now matches what Quran.com's web frontend does (`word_fields` query param). Once the next preview build lands and Mohammed flips the v2 toggle, pages should render with proper Mushaf glyphs.

- 2026-05-11 — Bismillah encoding fix (broken Arabic shaping on Baqarah opening + multi-surah pages)
  User reported broken Arabic rendering on Mushaf pages 2 (Baqarah opening) and 604 (Al-Ikhlas/Falaq/Nas) — letters disconnected, tashkeel floating in random positions, ligatures not forming. Page 1 (Al-Fatihah) rendered correctly. A settings toggle they flipped expecting a UI change did nothing. Root cause: the `BISMILLAH_UTHMANI` literal constant in `app/quran/mushaf.tsx` was encoded in **canonical Unicode NFC mark order** (fatha `U+064E` before shadda `U+0651`), while every Quranic text source we use (Tanzil, `assets/data/mushaf-pages.json`, Quran.Foundation API) uses the **Uthmani/Tanzil convention** — shadda first, then vowel mark. AmiriQuran's GSUB ligature rules are calibrated for the Uthmani order; when fed NFC-canonical input, Android's text shaper fails to form the shadda-over-letter composition. Letters render disconnected with combining marks in wrong positions. Page 1 wasn't affected because Fatihah's bismillah is verse 1 in the JSON (the source data is correct). Pages 2 and 604 render the constant in a separate `<Text>` element (Baqarah's bismillah row + the inline-bismillah path for multi-surah pages), hitting the broken encoding. Diagnosed via `scripts/probe-bismillah.mjs` which decoded both encodings codepoint-by-codepoint and surfaced the `064E 0651` vs `0651 064E` swap. Fix: re-encoded the constant to match the JSON byte-for-byte (also verified via the same probe). Added a comment in mushaf.tsx explaining the encoding choice, the failure mode, and pointing at the probe script. Added `scripts/verify-bismillah-encoding.mjs` as a CI tripwire that asserts the constant matches the JSON's canonical Uthmani encoding on every PR; wired into `.github/workflows/ci.yml` right after the existing critical-exports verifier. Catches the next regression where an NFC-normalising editor silently rewrites the mark order. Sentry breadcrumb added to `settingsStore.save()` — every settings change now records `key: previous → next` in the Sentry breadcrumb trail (excludes large blob keys like `cachedPrayerTimes`, dedups no-op writes, caps payload at 200 chars). When a user reports "I flipped X and nothing changed," the next Sentry issue we get from that session will show exactly which key and what direction it flipped. Lazy-required so Sentry misconfig can't break Settings. Settings toggle audit while I was in there: the only toggles in v1.7.7 that affect Quran reader rendering are `Quran font size` (slider — works), `Mushaf reading mode` (auto/light/sepia/dark — palette only, never affected Arabic shaping), `Show translation` / `Show transliteration` (surah-mode only, doesn't apply to Mushaf view). None of them would have "fixed" the broken bismillah — they're not wired to the Arabic encoding path, which is correct behaviour. The toggle Mohammed flipped most likely matched one of these. The new v1.8.0 "Use new renderer (Quran.Foundation)" toggle I added in commit `49914c2` only exists in the preview build (`in.deen.app.preview`); v1.7.7 doesn't have it. Verification: `npx tsc --noEmit` clean, `node scripts/verify-bismillah-encoding.mjs` green. Lands as part of the v1.8.0 preview build (build 2515c08f) — no separate hotfix release needed, since the v1.8.0 code_v2 + per-page font path will replace the literal-bismillah render entirely in the upcoming v1.8.0 ship.
